AI Agents App Review Brief

Live app: https://ai-agents-31pz.onrender.com/ | Updated: June 5, 2026

What Changed Since Last Review

  • The interface now has a ChatGPT-like layout layer: full-width chat, slide-out Agents/History drawer, right-side Tools/Settings drawer, compact composer, and source transparency styling.
  • The backend now supports shared AI accounts for all signed-in users through Render environment variables.
  • The provider adapter now attempts fallback providers instead of collapsing into a generic server error.
  • Supabase now has an ai_daily_usage table and increment_ai_daily_usage function prepared for per-user daily quota tracking.
  • The app still needs a final integration pass to enforce the daily request limit inside /api/chat/stream before model calls.

Current Architecture

Frontend

React + Vite SPA served by Express on Render. Static enhancement files are loaded from /chatgpt-layout.css and /chatgpt-layout.js.

Backend

Express server with authenticated API routes, streaming chat endpoint, upload handling, connectors, memory, and source trail generation.

Database

Supabase Auth and Postgres. User data is protected by RLS. Chats, messages, memory, uploads, connectors, and daily usage are stored per user.

Deployment

Render free web service. Current live URL is public, but users must sign in to use the workspace.

Shared Provider Fallback Plan

The desired behavior is that every signed-in user can ask agents questions without individually adding API keys. The backend keeps provider keys in Render env vars and never exposes them to the browser.

Gemini primary -> Groq fallback -> OpenRouter fallback -> Together fallback -> OpenAI fallback -> optional Ollama-compatible endpoint

Quota Question For Claude

If several friends use the same shared Gemini key, all users consume the same project quota. When the Gemini key hits rate limits or quota, the current adapter should try Groq/OpenRouter/Together/OpenAI if configured. If no fallback is configured, users receive a readable setup diagnostic instead of a blank/server error.

Please review: Is this fallback order sensible for a small public multi-agent workspace? Would you recommend a different order, a circuit breaker, or provider health cache?

Daily Usage Guard

Supabase migration already added:

public.ai_daily_usage(user_id, usage_date, request_count, token_estimate)
public.increment_ai_daily_usage(target_user_id, request_increment, token_increment)

The next implementation should call this function before each model request and block or downgrade requests when limits are exceeded.

Proposed policy

  • Free/shared users: 30 agent requests per day.
  • Soft token estimate: 60,000 characters per day, approximate until real token usage is tracked.
  • When a user reaches the limit: allow them to add their own key or try again tomorrow.
  • Admin/owner email can have a higher limit through an env allowlist.

Known Risks

  • Shared provider abuse: One user can consume the shared quota unless daily limits are enforced in chat route.
  • Provider health: Repeatedly trying a failed provider can slow every response. A short-lived failure cache may be needed.
  • Source quality: Source trail exists, but source cards still need date, domain type, and excerpt quality scoring.
  • Coder safety: Coder agent still needs explicit Accept/Reject confirmation for file writes and shell commands before it becomes autonomous.
  • Render free tier: Cold starts will delay first response. Frontend could move to Cloudflare Pages while keeping Express API on Render.
